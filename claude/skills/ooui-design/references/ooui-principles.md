# OOUI の核となる原則

上野学『オブジェクト指向UIデザイン』の考え方を、このスキルの実務ルールに落としたもの。
すべてのフェーズでこれらを判断基準にする。

## 1. オブジェクト主導で組み立てる

タスク指向UI（やりたいこと＝動詞を起点に画面を作る）は、操作のたびに「何をするか」を選ばせ、対象が後回しになる。
OOUI は逆に、ユーザーがまず **操作対象（オブジェクト＝名詞）** に出会い、それを選んでから操作を選ぶ。

- 悪い例（タスク指向）：トップに「新規登録」「検索」「変更」「削除」というメニューが並ぶ。
- 良い例（オブジェクト指向）：トップに「案件」「顧客」「見積」という対象が並び、各対象の一覧・詳細から操作する。

## 2. 「対象 → 操作」の順序

ユーザーの認知の流れを「対象を選ぶ → できる操作が見える → 操作する」に合わせる。
「操作を選ぶ → 対象を選ばされる」というモーダルな流れ（ウィザード的拘束）は、必要な場合を除き避ける。

## 3. コレクションビューとシングルビュー

主オブジェクトは原則として2種のビューを持つ。
- **コレクションビュー**：そのオブジェクトの集合（一覧）。絞り込み・並べ替え・新規作成の起点。
- **シングルビュー**：単一インスタンスの詳細。属性の表示・編集、関連オブジェクトへの導線、個別アクション。

## 4. アクションはオブジェクトに帰属させる

「編集画面」「承認画面」といった動詞ベースの画面を量産しない。
編集・承認・複製などのアクションは、対象オブジェクトのシングル／コレクションビュー上の操作として設計する。

## 5. 名詞と動詞を分離する

- 名詞（オブジェクト）：案件、顧客、見積、担当者 … UIの構造を決める。
- 動詞（タスク／アクション）：登録する、承認する、割り当てる … オブジェクトに紐づく操作になる。

タスク抽出時に出てきた動詞は、最終的にどれかのオブジェクトのアクションへ吸収される。どのオブジェクトにも属さない動詞が残ったら、オブジェクトの抽出漏れを疑う。

## 6. モードレスを志向する

ユーザーがいつでも対象を切り替えられ、操作の途中でも別の対象を見られる状態を目指す。
モード（同じ操作が状態によって違う結果になる仕組み）は極力減らす。編集／閲覧モードのトグル、ウィザード、モーダルダイアログの常用を避け、選択 → 操作・インライン編集・ノンブロッキングで置き換える。
モーダルは、不可逆操作の確認や本質的に逐次必須な手続きに限定する。
具体的な手法・検知リスト・批評観点は `references/modeless-design.md` を参照。

## アンチパターン検知

設計中に次が出たら立ち止まり、オブジェクト指向の代替案を提示する。
- ルートナビが動詞（「登録」「検索」「集計」）で構成されている。
- 同じオブジェクトに対し「一覧画面」と無関係に「編集画面」「詳細画面」が乱立している。
- どのオブジェクトにも属さない独立画面が多い。
- 1画面に複数オブジェクトの操作を詰め込み、主役が不明瞭。
