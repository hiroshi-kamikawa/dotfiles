# フェーズ4：ナビゲーション設計

## ゴール

どのオブジェクトが **ルート（トップ）から直接たどれるか**、そして **オブジェクト間をどう行き来するか** を決め、図にする。
細かいUIには立ち入らない。画面の存在と遷移の構造だけを決める。

## 進め方

1. `03-concept-model.vN.md`（最新版）を読む。
2. **ルートナビゲーションを決める**：トップから一覧（コレクション）に直接アクセスできる主オブジェクトを選ぶ。
   - 多くは利用頻度が高く独立性の高いオブジェクト（例：案件、顧客）。
   - 他オブジェクトに従属する弱いオブジェクト（例：見積は案件配下）はルートに置かず、親のシングルビューから辿らせる、という判断もある。
   - **ルートを動詞で構成しない**（原則1）。あくまでオブジェクトの集合を起点にする。
3. **オブジェクト間遷移を設計する**：あるオブジェクトのシングルビューから、関連する別オブジェクトのコレクション／シングルへどう辿るかを決める（概念モデルの関連を導線に変換）。
4. コレクションビュー → シングルビュー → 関連オブジェクトのコレクション、という基本の入れ子を意識する。

## 可視化（Mermaid graph）

ノード＝画面（コレクション／シングル）、エッジ＝遷移。

```markdown
## ナビゲーション図

\`\`\`mermaid
graph TD
  ROOT[トップ]
  ROOT --> 案件一覧[案件: コレクション]
  ROOT --> 顧客一覧[顧客: コレクション]
  案件一覧 --> 案件詳細[案件: シングル]
  案件詳細 --> 見積一覧[見積: コレクション（案件配下）]
  見積一覧 --> 見積詳細[見積: シングル]
  顧客一覧 --> 顧客詳細[顧客: シングル]
  顧客詳細 --> 案件一覧
\`\`\`
```

## モードレスの観点

`references/modeless-design.md` を参照。ナビでは特に、ウィザード的な強制遷移（前へ・次へで段階を縛る流れ）を避け、オブジェクト間を自由に行き来できる構造にする。ある作業の途中でも別オブジェクトを見に行ける導線を確保する。

## 出力フォーマット

```markdown
- 理由: …

## 従属オブジェクト
- 見積: 案件のシングルビューから辿る

## ナビゲーション図
<上記 graph>

## 主な遷移
| 起点 | 遷移先 | きっかけ |
|------|--------|----------|
| 案件一覧 | 案件詳細 | 行を選択 |
| 案件詳細 | 見積一覧 | 関連タブ「見積」 |
```
