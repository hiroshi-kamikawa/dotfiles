# フェーズ6：ワイヤーフレーム

## ゴール

ビュー設計をもとに、**クリックで画面遷移できる低精度ワイヤーフレーム** を1枚のHTMLで作る。
あくまでラフ。色やブランド、細かいレイアウトは作り込まない。グレースケールの箱とラベルで、構造と遷移が伝わればよい。

## 方針

- `assets/wireframe-template.html` を雛形にする。
- **自己完結した単一HTMLファイル**にする（外部依存なし、ローカルでそのまま開ける）。
- 画面の切り替えは **ハッシュ（#screen-id）ベースのSPA的な遷移** で実装する。一覧の行や、詳細内のボタン・関連リンクをクリックすると対応画面に切り替わる。
- 各画面は `05-views.vN.md` の「表示要素・操作・遷移先」を反映する。
  - コレクション：簡単なテーブル/リスト。行クリックでシングルへ。「新規作成」ボタンを置く。
  - シングル：プロパティの箱、関連コレクションの埋め込み、アクションボタン。ボタンは遷移先があれば遷移、なければ無反応（ラフなので可）。
- 上部に画面名、戻る導線、ルートナビ（主オブジェクトの一覧へのリンク）を置く。

## モードレスの表現

`references/modeless-design.md` を参照。ラフでも、モードレスな相互作用が伝わるように示す。
- 別画面の「編集モード」ではなく、シングルビュー内での **インライン編集** をラフに示す。
- 行や項目を選ぶと出る **アクションバー**（選択 → 操作）で表現する。
- モーダルは削除確認など正当な箇所のみ。常用しない。

## 低精度を保つ

- 実データやスタイリングに凝らない。`border`, `padding`, グレー背景程度。
- 画像・アイコンは不要。プレースホルダの箱で示す。
- 目的は「どの画面に何があり、どう繋がるか」をクリックで体感できること。

## 出力

`docs/ooui/<app-name>/wireframes/index.html`（再実行時は `wireframes/index.v2.html` のように版を分ける）。

## 生成方式：コピー＋差分適用（重要）

HTMLが大きくなると、一括の `Write` は途中でストリームが停止しやすい（`API Error: Response stalled mid-stream.`）。これを避けるため、**前版がある場合は「まるごと書き直し」ではなく「コピー → 差分適用」で作る**。

**初回生成（前版なし）**
- `assets/wireframe-template.html` を土台にする。
- まずテンプレートを新規ファイルにコピーする（例: `cp assets/wireframe-template.html docs/ooui/<app-name>/wireframes/index.html`。テンプレートのパスはスキルディレクトリ配下）。
- そのうえで `Edit` を使い、画面（screen）を1つずつ追記・肉付けしていく。1回の `Edit` で全画面を入れようとせず、**画面単位・セクション単位に分けて**適用する。

**再実行・改善（前版あり）**
- 新版をゼロから `Write` しない。**直近の版をコピーしてから差分だけ `Edit` で当てる**。
  1. `cp docs/ooui/<app-name>/wireframes/index.html docs/ooui/<app-name>/wireframes/index.v2.html`（版を上げてコピー）。
  2. コピー先を `Read` して現状を把握する。
  3. 変わった画面・要素だけを `Edit`（必要なら `replace_all`）でピンポイントに書き換える。上流（ビュー／ナビ／概念モデル）の変更点に対応する箇所のみを触る。
- 変更が広範でも、**1つの巨大 `Edit`／`Write` にまとめない**。画面ごと・関連ブロックごとに小さな `Edit` を積み重ねる。

**大きくなってきたときの指針**
- 1ファイルが肥大化して `Edit` でも扱いづらい場合は、共通の CSS / ナビ枠と各画面を分割し、画面追加は追記ベースで行う。
- どうしても全面刷新が必要なときも、空ファイルへ一気に `Write` するのではなく、骨組みを `Write` → 画面を `Edit` で順次追加、と段階に分ける。

## 完了時

`references/critique-and-iteration.md` の完了プロトコルに従う。
- 提示: ファイルパスを伝え、ブラウザで開いて確認してもらう
- 特に見る批評観点: クリック遷移が破綻なく通るか、画面の過不足、空状態・エラーの表現方針、アクセシビリティの基礎（コントラスト・キーボード・ラベル）
- 反復ゲート: 「設計のどこを直すか（＝どのフェーズに戻るか）」を確認し、ループに入る。満足するまで磨き続ける。

## ループ時の注意

- ビュー／ナビ／概念モデルを直したら、ワイヤーフレームは陳腐化する。`00-index.md` のフラグに従い、再生成する。
- 上流の最新版を必ず読み直してから作り直す。
- 再生成は「コピー＋差分適用」で行う（上記「生成方式」参照）。前版をコピーして版を上げ、上流の変更に対応する画面・要素だけを `Edit` で当てる。全面 `Write` はストリーム停止の原因になるため避ける。
