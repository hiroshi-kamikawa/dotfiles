# OOUI デザイン

業務アプリケーションをゼロから設計するための、OOUI（オブジェクト指向UIデザイン）ワークフロー。
ユーザーと対話しながら、タスク（ユースケース）からオブジェクトを抽出し、概念モデル・ナビゲーション・ビューへと落とし込み、最後にクリック可能なワイヤーフレームまで作る。

**ユーザー向けの応答・成果物はすべて日本語で書く。**

---

## このスキルの貫く原則（OOUI の核）

設計のあらゆる段階で次を守る。詳細は `references/ooui-principles.md`、モードレス設計の具体は `references/modeless-design.md` を参照。

1. **オブジェクト主導**。先にタスク（動詞）を集めるが、UIはタスク（動詞）ではなくオブジェクト（名詞）を中心に組み立てる。
2. ユーザーは **「対象を選んでから操作を選ぶ」**。操作を選んでから対象を選ばせるモーダルな流れは避ける。
3. すべての主オブジェクトは原則 **コレクションビュー（一覧）とシングルビュー（単一）** を持つ。
4. アクションは画面ではなく **オブジェクトに帰属** させて設計する。
5. 名詞（オブジェクト）と動詞（タスク）を混同しない。
6. **モードレスを志向する**。隠れたモード（編集／閲覧トグル、ウィザード、常用モーダル）を減らし、選択→操作・インライン編集・ノンブロッキングで置き換える。

これらに反する設計案が出たら、ユーザーに代替案を示して軌道修正する。

---

## 基本姿勢：対話・専門批評・反復

このスキルは **1発で完成させるものではない**。ユーザーが満足するまで、各段階を行き来して磨き続けるためのもの。
詳細な進め方とチェック観点は `references/critique-and-iteration.md` を参照。各フェーズで常に次の3つを行う。

1. **深掘り質問**：成果物を確定する前に、ユーザーの意図・制約・優先度を引き出す。一度に詰め込まず、要点を絞って対話的に問う。決めつけずに選択肢を提示し、ユーザーの判断を仰ぐ。
2. **専門家としての批評**：作った成果物に対し、**OOUI方法論家** と **UI/UXのプロのデザイナー** の二つの視点でフィードバックする。良い点・リスク・具体的な代替案とトレードオフを、忖度せず、しかし押し付けず示す（観点は `references/critique-and-iteration.md`）。
3. **反復ゲート**：どのフェーズも「完了＝次へ強制」にしない。「ここを直す／このまま次へ」をユーザーに委ね、満足するまで同じフェーズに留まってよい。成果物は常に下書きとして扱う。

## 成果物と版管理

すべての成果物は次の場所に置く（`<app-name>` は対象アプリのスラッグ）。

```
docs/ooui/<app-name>/
  00-index.md            # 現状サマリ・各成果物の最新版リンク・版履歴・陳腐化フラグ
  01-usecases.v1.md      # フェーズ1：ユースケース（タスク）一覧
  02-objects.v1.md       # フェーズ2：オブジェクト抽出
  03-concept-model.v1.md # フェーズ3：概念モデル（Mermaid classDiagram）
  04-navigation.v1.md    # フェーズ4：ナビゲーション設計（Mermaid graph）
  05-views.v1.md         # フェーズ5：ビュー設計
  wireframes/index.html  # フェーズ6：クリック遷移できるワイヤーフレーム
  as-is/                 # 既存モード：現状の再構成（objects/navigation/views の as-is 版）
  audit.v1.md            # 既存モード：監査結果（指摘・重大度・改善案）
```

（`as-is/` と `audit.v1.md` は既存プロジェクト分析・改善モードでのみ作る。詳細は `references/existing-project-audit.md`。）

**版管理ルール（ファイル名方式）**
- ファイルは上書きしない。フェーズを再実行したら新しい版 `vN`（v2, v3, …）を作る。
- 旧版は残す。`00-index.md` に最新版へのリンクと版履歴を記録する。
- 上流フェーズを更新したら、下流の成果物に **陳腐化フラグ** を立てる（例：概念モデルを v2 にしたら、ナビ／ビュー／ワイヤーは「要再検討」）。`assets/index-template.md` を雛形に使う。
- 各フェーズの実行時は、上流ドキュメントの **最新版** を読んでから始める。

## 実行の流れ（ルーティング）

### 0. モードと対象の特定

まず、どちらのモードかを見極める（不明ならユーザーに確認）。

- **新規設計モード**：ゼロから業務アプリを設計する。下記フェーズ1から進める。
- **既存プロジェクト分析・改善モード**：既にコード／UIがあり、OOUI・モードレス観点でフィードバック（必要なら改善設計）したい。このモードでは `references/existing-project-audit.md` に従う（現状把握 → As-isモデル化 → 監査 → 必要なら改善設計）。

次に対象アプリを特定する。`docs/ooui/` を確認し、
- 既存の設計ドキュメント `<app-name>/` があれば `00-index.md` を読んで現状（最新フェーズと各版）を把握する。
- 無ければアプリ名（スラッグ）と一言の目的をユーザーに聞き、ディレクトリと `00-index.md` を作る。

### 1. 入口の決定（通し or フェーズ指定の両対応）

- ユーザーがフェーズを指定（例「ビュー設計だけやり直したい」）→ そのフェーズへ直行する。
- 指定がない → `00-index.md` の現状から、続きのフェーズを提案して通しで進める。

### 2. フェーズの実行

下表のフェーズを順に進める。各フェーズの **詳細手順は必ず該当の参照ファイルを読んでから** 実行する。

| # | フェーズ | 参照ファイル | 主な成果物 | 可視化 |
|---|---|---|---|---|
| 1 | ユースケース抽出 | `references/phase1-usecases.md` | タスク一覧（対象＋動詞） | 表 |
| 2 | オブジェクト抽出 | `references/phase2-objects.md` | オブジェクト一覧 | 表 |
| 3 | 概念モデル | `references/phase3-concept-model.md` | オブジェクト＋属性＋アクション＋関連（多重度） | Mermaid classDiagram |
| 4 | ナビゲーション設計 | `references/phase4-navigation.md` | ルートナビ＋オブジェクト間遷移 | Mermaid graph |
| 5 | ビュー設計 | `references/phase5-views.md` | 画面ごとの表示要素・操作・遷移 | 表＋Mermaid |
| 6 | ワイヤーフレーム | `references/phase6-wireframe.md` | 低精度・クリック遷移可能なラフ | HTML |

### 3. フェーズ完了時のお作法（完了プロトコル）

各フェーズが終わったら必ず、`references/critique-and-iteration.md` の完了プロトコルに従う。要約：
1. 成果物ファイル（`0N-*.vN.md` 等）を書き出す。
2. `00-index.md` を更新する（最新版リンク・版履歴・下流の陳腐化フラグ）。
3. ユーザーに **要点サマリ＋可視化** を提示する。
4. **専門批評**：OOUI と UI/UX プロの視点で、良い点・リスク・代替案を2〜3点提示する。
5. **深掘り質問**：意図・制約・優先度を引き出す問いを投げる。
6. **反復ゲート**：「ここを直す / このまま次へ」をユーザーに委ねる。満足するまで先へ進めない。

## ループ改善

OOUI は行き来しながら洗練させる前提。次を徹底する。

- **「完成」という終点は無い。基準はユーザーの満足**。各フェーズで深掘り質問と専門批評を重ね、納得いくまで磨く。
- どのフェーズからでも再入できる。再実行時は新版 `vN` を作る。
- 上流を変えたら、下流の整合性を確認し、ズレを `00-index.md` の陳腐化フラグとサマリに明示する。
- 「対象を選んでから操作」「コレクション＋シングル」「アクションはオブジェクトに帰属」を、各再検討時にも点検する。

---

## 進め方のスタイル

- 一度に大量に質問しない。要点を絞って対話的に進める。
- ユーザーが概念に不慣れな様子なら用語を一言で補う。詳しいなら冗長な説明は省く。
- 各フェーズの粒度は、参照ファイルの指示に従う。ビュー／ナビは「どの画面に何を出すか」までで、細かいUIには踏み込まない（ワイヤーフレームで初めて画面の形を示す）。
