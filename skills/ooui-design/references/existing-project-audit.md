# 既存プロジェクトの分析・改善（既存モード）

ゼロからではなく、すでにコード／UIがあるプロジェクトに対し、現状を OOUI・モードレス観点で評価し、フィードバック（必要なら改善設計）につなげるモード。
Claude Code のファイルアクセスで実コードを読めることが前提。

## 全体の流れ

1. **現状把握（リバースエンジニアリング）**：プロジェクトを読み、現行の画面・ルート・オブジェクト・データモデルを棚卸しする。
2. **As-is モデル化**：現状を本スキルの成果物形式で「現状版（as-is）」として再構成する。
3. **監査（audit）**：OOUI原則・モードレス・UI/UX観点に照らし、良い点／問題点／重大度／根拠を整理する。
4. **改善（to-be）**：必要なら通常フェーズで理想形を設計し、差分と移行方針を示す。

ユーザーが「フィードバックだけ欲しい」なら 3 まで。「直したい」なら 4 へ進む。最初に意図を確認する。

---

## 1. 現状把握（リバースエンジニアリング）

フレームワークに合わせて、次を静的に読む（ビルドや実行は不要）。

- **ルーティング定義** → 画面とURL構造。例：Next.js の `app/` や `pages/`、React Router 設定、各種ルータ定義。
- **画面・コンポーネント** → 各画面が何を表示し、どの操作（ボタン・アクション）があるか。
- **データモデル／スキーマ** → オブジェクト候補。DBマイグレーション、ORMモデル、型定義、APIスキーマなど。
- **ナビ／メニュー定義** → ルートナビ構成。

読みながら **推定の根拠（ファイルパス）** を必ずメモする。コードから読めた事実と、自分の解釈・推測を区別する。不明点は憶測で埋めず、ユーザーに確認する。

効率重視：大規模コードを全部読まない。**ルーティング → 主要画面 → データモデル** の順で骨格を掴む。

## 2. As-is モデル化

`docs/ooui/<app>/as-is/` 配下に、現状を「今あるものの正確な写し」として再構成する（理想を混ぜない）。

- `objects.as-is.v1.md`：推定オブジェクトと由来ファイル。
- `navigation.as-is.v1.md`：現行のルート・遷移（Mermaid graph）。
- `views.as-is.v1.md`：画面ごとの表示要素・操作。

形式は通常フェーズ2・4・5の出力に準じる。

## 3. 監査（audit）

`docs/ooui/<app>/audit.vN.md` を作る。`references/ooui-principles.md`・`references/modeless-design.md`・`references/critique-and-iteration.md` の観点で評価する。

各指摘は次の構造で書く。

| 重大度 | 観点 | 現状（根拠ファイル） | 問題 | 改善案（トレードオフ） |
|--------|------|----------------------|------|------------------------|

- **重大度**：高／中／低。
- **観点**：OOUI／モードレス／情報設計／ナビ／状態とフィードバック 等。
- 良い点も挙げる。忖度せず、しかし押し付けず。重大度順に並べる。

代表的な検知パターン：
- ルートナビが動詞で構成されている（タスク指向）。
- あるオブジェクトにコレクション／シングルの一方が欠落している。
- 動詞ベースの独立画面が乱立している。
- 隠れたモード、編集／閲覧トグル、常用モーダル・ウィザード。
- 1画面に主役オブジェクトが不明瞭（複数オブジェクトの操作が混在）。
- 同じ概念が画面ごとに別の用語で呼ばれている（一貫性欠如）。

監査のサマリ（重大度別の件数・要点）を `00-index.md` にも反映する。

## 4. 改善（to-be）※ユーザーが希望する場合

- 通常フェーズ（1 → 6）を、as-is を入力にして実行する。to-be 成果物は通常の番号付きファイル（`01-…` 〜、`wireframes/`）。
- **既存の制約を深掘り質問で引き出す**：技術スタック、データ移行、後方互換、工数、リリース都合。これを踏まえて現実的な改善にする。
- as-is → to-be の **差分と段階的移行方針**（どこから着手すると効果が大きいか、リスク、影響範囲）をまとめる（`audit.vN.md` 末尾か別ファイル）。
- 「全面刷新」だけでなく「低コストで効く部分改善」も提示し、ユーザーに選ばせる。

## 注意

- 推測を断定しない。読めた事実と解釈を分けて書く。
- 監査は判断材料の提供。既存設計者の意図や制約を尊重し、決定を上書きしない。
- このモードも反復前提。監査も改善案も、ユーザーが納得するまで深掘り質問と批評を重ねる。
