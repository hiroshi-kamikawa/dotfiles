# Repository Quality Guard

## このスキルについて

`hk-quality-guard-checker` は、対象リポジトリを診断し、そのプロジェクトに合った品質チェックの配置を設計・導入・修復するためのスキルです。Huskyを導入すること自体を目的にはせず、Gitフック、stagedファイルの振り分け、手動実行できる共通コマンド、実際の検査ツール、CIを分離します。CIはワークフローの存在とマージ強制状態を分けて評価し、ブランチ保護や必須チェックを確認できた場合だけ「回避不能」と判断します。

既存のLint、Formatter、型チェック、テスト、ビルド、アクセシビリティ検査、秘密情報検査、依存関係監査などを優先して再利用します。別のフック管理ツールがある場合や、既存CIへ大きな影響が出る場合は、自動的に置換せず競合と選択肢を報告します。

## 効果的な使い方

調査だけなら「このリポジトリの品質担保をauditして」、最低限の導入なら「setup-minimalで品質チェックを入れて」、広い品質保証なら「setup-fullで提案して」のように依頼します。モードを指定しなくても、依頼内容から変更範囲の小さいモードが選ばれます。

導入を依頼する際は、許容できるpre-commit時間、CIの実行時間や料金、コミットメッセージ規約の有無、秘密情報検査ツールの管理方法などを伝えると判断が安定します。新規依存関係や規約を追加する前には変更案が提示されるため、チーム事情に合わない項目を除外できます。

## 実装の背景と決定

ローカルフックは無効化できるため、それだけで品質を保証する設計にはしていません。フック管理ツールは実行タイミング、staged-fileツールは対象ファイルの選択、package scriptsやタスクランナーなどのプロジェクトコマンドは手動実行と再利用の窓口、専門ツールは実検査、CIはマージ保証の候補という責務に分けています。CIを強制層と判断するには、プロバイダー側の必須チェックとブランチ保護も確認します。

`SKILL.md` は診断から報告までの安全な進行を定義し、詳しい判断表は `references/` に分離しました。`scripts/` の診断ツールは依存パッケージなしで動作し、Node.js、Python、PHP、Go、Rust、Rubyのルートおよびネストしたマニフェスト、サブプロジェクトごとの依存管理、再帰的に見つけた品質設定、CIの推測結果をJSONとして返します。探索には上限があり、打ち切られた場合は結果に記録されます。機械診断だけで変更を決めず、必ず元の設定ファイルとコマンドを確認する設計です。

pre-commitにはstaged対象の高速処理、pre-pushには型チェックやUnit Testなどの中量処理、CIにはビルドやE2Eを含む全体検査を置きます。全体Format、新しいコミット規約、厳しい閾値、既存ツールの移行は大きな差分や運用変更につながるため、明示的な承認なしには行いません。

## スキル自体の検証

診断スクリプトの回帰テストは追加依存なしで実行できます。

```sh
node --test skills/hk-quality-guard-checker/tests/*.test.mjs
```

複数言語とネストしたマニフェスト、Node.jsのロックファイル競合、CI強制状態、要約出力、Gitフック管理の競合をフィクスチャで検証します。
